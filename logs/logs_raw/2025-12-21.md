## ① Physon React 前提：一段深いシステム処理概要（文章版）

### フロントエンド（Physon React / React）

**主な責務**
- 画面描画
- 状態管理
- API 通信
- 入力検証（軽量）

**内部構成（深掘り）**
```
[UI Components]
  ├─ Page Components
  │    ├─ 画面単位（一覧・詳細・編集）
  │
  ├─ Presentational Components
  │    ├─ Button / Form / Table
  │
  ├─ Hooks
  │    ├─ useState / useEffect
  │    ├─ useCustomHook（API呼び出し）
  │
  ├─ State管理
  │    ├─ Context / Redux / Zustand 等
  │
  └─ API Client
       ├─ fetch / axios
       ├─ 認証トークン付与
       └─ エラーハンドリング
```

**通信フロー**
```
ユーザー操作
 → 状態更新
 → API Client
 → HTTP Request（JSON）
```

### バックエンド（Python）

**処理レイヤー分割（重要）**
```
[API Layer]
  ├─ ルーティング

  │
[Service Layer]
  ├─ 業務ロジック
  ├─ トランザクション制御
  │
[Domain / Model Layer]
  ├─ Entity
  ├─ DTO
  │
```

**処理フロー（深掘り）**
```

④ RepositoryでDB操作
⑤ 結果をJSON化
⑥ フロントへレスポンス
```

## ② REST API と FastAPI（fast）の違い

ここ、**よく誤解されやすいポイント**です。

### 結論を先に
👉 **RESTは設計思想**  
👉 **FastAPIは実装フレームワーク**

つまり **比較対象がズレている**のが本質です。

---

## ③ REST API とは（思想・ルール）

**REST = APIの設計ルール**

### RESTの特徴
- HTTPメソッドを使い分ける
- URLは「リソース」を表す
- ステートレス

例：
```
GET    /users        → 一覧取得
GET    /users/1      → 取得
POST   /users        → 作成
PUT    /users/1      → 更新
DELETE /users/1      → 削除
```

RESTは  
✅ Flaskでも  
✅ FastAPIでも  
✅ Djangoでも  
**実装できる**

---

## ④ FastAPI とは（実装）

**FastAPI = PythonのWeb APIフレームワーク**

### FastAPIの特徴（深掘り）
- 非同期（async/await）対応
- 型ヒント必須
- 自動Swagger生成
- 高速（Starlette + Pydantic）

```python
@app.get("/users/{id}")
async def get_user(id: int):
    return {"id": id}
```

### FastAPIは
- **REST APIを作るための道具**
- RESTでないAPIも作れる

---

## ⑤ Flask（REST） vs FastAPI（REST）の実務比較

| 観点 | Flask | FastAPI |
|----|----|----|
| 学習コスト | 低い | やや高 |
| 型安全 | 弱い | 強い |
| 非同期 | 弱い | 強い |
| 自動ドキュメント | 手動 | 自動 |
| 大規模向き | △ | ◎ |

---

### 中〜大規模 / 将来拡張

- **REST** → APIの「作り方のルール」
- **FastAPI** → RESTを「実装するための高速な道具」
- **比較するなら**
  - Flask vs FastAPI
  - REST vs GraphQL

- エラーハンドリング共通化

👉 **React と FastAPI の境界線**

---


**責務**
- URL
- HTTP Method
- 入出力定義

**やらない**
- 業務判断
- DB操作

👉 **Controller的役割**

---

```python
class UserCreate(BaseModel):
```

- 型
- 必須
- フォーマット

👉 **FastAPIが強い理由**

---

```python
def create_user(data):
```

**責務**
- 業務ロジック

- トランザクション

👉 **ここがシステムの心臓**

### ⑥ Repository Layer

```python
def insert_user(session, user):
```

**責務**
- DBアクセスのみ
- SQL / ORM操作

👉 **業務ロジック禁止**

```

 ├─ pages/        ← 画面（ルーティング単位）
 ├─ components/   ← 見た目だけ
 ├─ hooks/        ← ロジック
 ├─ api/          ← 通信
 ├─ stores/       ← 状態管理
 ├─ types/        ← 型
 └─ utils/        ← 汎用処理
```

### 📄 pages（画面）

**責務**
- 画面構成
- hooks の呼び出し

**命名**
```

```

**禁止**
- API直接呼び出し
- 業務ロジック

👉 *Pageは「組み立て役」*

### 🧩 components（UI部品）

**責務**
- 表示
- onClick 等の通知

**命名**
```

```

**禁止**
- API
- useEffect
- 状態管理

👉 *componentsは「見た目専用」*

### 🪝 hooks（ロジック）

**責務**
- 状態
- API呼び出し
- データ加工

**命名ルール（厳守）**
```

useCreateUser.ts
```

```ts
export const useUserList = () => {}
```

👉 **use + 名詞**以外はNG  
👉 ここが「ReactのService層」

**責務**

**命名**

```ts
getUsers()
createUser()
```

👉 画面ロジック禁止  
👉 state操作禁止


**禁止**
- SQL
- HTTP依存

👉 *ここがシステムの本体*

**責務**

**命名**
```
user_repository.py
```

```python
insert_user()

```

**禁止**
- if文で業務判断
- 画面意識

# 処理概要（設計用・3物理サーバ構成）

## 1. 全体構成方針（設計思想）

- 物理サーバは **3層分離**

  - エンド（アプリ）サーバ
  - データベースサーバ
- **処理責務を厳密に分離**
- 業務ロジックは **エンドサーバに集約**
- DBは **状態管理（永続化）専用**

---

## 2. 物理サーバ別 処理責務定義

---

### ① フロントサーバ（Front Server）

**役割**
- 画面表示
- ユーザー操作の受付
- API呼び出し

**搭載コンポーネント**
- React（Physon React）
- HTML / CSS
- JavaScript / TypeScript
- Node.js（ビルド・配信）

**処理内容（設計記載例）**
- 画面レンダリング
- フォーム入力制御
- クライアント側バリデーション
- APIリクエスト生成（fetch / axios）

**非責務（明示）**
- 業務ロジックを持たない
- DBに直接アクセスしない

---

### ② エンドサーバ（Application / End Server）

**役割**
- 業務処理の中核
- セキュリティ境界

**搭載コンポーネント**
- IIS（リバースプロキシ）
- Python / Flask
- WSGI
- ORM（SQLAlchemy）

**処理内容（設計記載例）**
- 認証・認可（JWT）
- REST APIルーティング
- 業務ロジック実行
- データ整形・変換
- トランザクション制御
- エラーハンドリング
- ログ出力

**責務の強調（レビュー対策）**
> 本システムにおけるすべての業務判断はエンドサーバで実施する

**役割**
- データ永続化
- 整合性担保

**搭載コンポーネント**

**処理内容（設計記載例）**

（別紙第１０の２） 、品目等内訳書（乙） （別紙第１０の３） 、発注依頼書（一般） （別
紙第１０の４）及び品目等内訳書（発注） （別紙第１０の５）により行うものとする。
（納入糧食の検査）
第３３条 納入される糧食は、まず食品衛生管理官が食品衛生検査を実施し、これに合
格したものについて、糧食の検査官が受領検査を実施するものとする。
第４節 食事の配分等
（食事の配分）
第３４条 食事は、通常、食堂において各人ごとに直接配分する。ただし、施設の状況
その他により食堂以外の場所において配分し、又はその代表者に一括して配分するこ
とができる。
第３５条 削除
（現品の配分）
第３６条 現品は、中隊長等に一括して配分する。
２ 中隊長等は、演習等の計画変更により、現品に未使用品を生じたときは、給食実施
機関の長の定めるところに従い、返納しなければならない。
（有料支給の集計等）
第３７条 有料支給の集計については、 有料支給内訳表 （別紙第１１） で行うものとし、
有料支給内訳表は、債権発生通知書（防衛省債権管理事務取扱細則（平成１８年防衛
庁訓令第１０７号）第５５条に規定する書面をいう。以下同じ。 ）の証拠書類として
保存する。
（検食等）
第３８条 給食実施機関の長は、検食所見簿（給食実施機関の長） （別紙第１２）及び
検食所見簿（食品衛生管理官） （別紙第１２の２）を備え付け、検食を実施した者の

３ 給食実施機関の長（その権限の委任を受けた者を含む。 ）は、行動等のため駐屯地

第４１条 削除

第６節 部外者の給食
（部外者の給食）
第４３条 部外者への給食は、 駐屯地における給食又は運搬食 （以下 「給食等」 という。 ）
を原則とし、非常用糧食及び戦闘糧食Ⅱ型（以下「非常用糧食等」という。 ）は、部
外者に支給しないものとする。ただし、省令第１４条に規定する災害救助その他の行
動等において給食等が困難な場合又は訓令第２５条に規定する教育訓練若しくは自
衛隊の視察若しくは見学において部外者に非常用糧食等を支給する場合は、この限り
ではない。
（食事の配分等）
第４３条の２ 給食担当官は、食券を確認の上、食事を配分するものとし、回収した食
券は総務科長等に送付するものとする。
２ 総務科長等は、給食担当官から送付された食券を点検し、食需伝票（別紙第２）と
相違のないことを確認した後、債権発生通知書の証拠書類として保存するものとする。
（食事代の徴収）
第４４条 部外者が食事の申込みをしたものについては、第４０条の規定を準用する。

実施機関を指定し、その給食実施状況の実態を調査する。

の給食審査を行う。
３ 陸上幕僚長は、前２項の結果に基づき改善すべき事項について、方面総監等に指示
するものとする。

from pptx import Presentation
from pptx.util import Inches
from pptx.enum.shapes import MSO_AUTO_SHAPE_TYPE
from pptx.enum.shapes import MSO_CONNECTOR

prs = Presentation()
slide = prs.slides.add_slide(prs.slide_layouts[6])

# Title
title = slide.shapes.add_textbox(Inches(0.5), Inches(0.2), Inches(9), Inches(0.6))
title.text_frame.text = "陸上自衛隊 給食業務システム概要図（1枚構成）"

def add_box(left, top, width, height, text):
    shape = slide.shapes.add_shape(
        MSO_AUTO_SHAPE_TYPE.ROUNDED_RECTANGLE,
        left, top, width, height
    )
    shape.text_frame.text = text
    return shape

# Main vertical flow
box1 = add_box(Inches(3.5), Inches(1.2), Inches(3), Inches(0.9), "陸上幕僚長\n給食業務の統制")
box2 = add_box(Inches(3.5), Inches(2.4), Inches(3), Inches(0.9), "方面総監 等\n給食要領策定・審査")
box3 = add_box(Inches(3.5), Inches(3.6), Inches(3), Inches(0.9), "給食実施機関の長\n（駐屯地）")
box4 = add_box(Inches(3.5), Inches(4.8), Inches(3), Inches(0.9), "給食担当官\n給食管理・調理実施")

# Side roles
box5 = add_box(Inches(0.5), Inches(4.8), Inches(2.7), Inches(0.9), "栄養担当官\n献立作成・栄養管理")
box6 = add_box(Inches(6.8), Inches(4.8), Inches(2.7), Inches(0.9), "食品衛生管理官\n衛生検査・検食")

# System
box7 = add_box(Inches(3.5), Inches(6.1), Inches(3), Inches(0.9), "陸自業務システム\n糧食管理機能")

def connect(from_shape, to_shape):
    connector = slide.shapes.add_connector(
        MSO_CONNECTOR.STRAIGHT,
        from_shape.left + from_shape.width/2,
        from_shape.top + from_shape.height,
        to_shape.left + to_shape.width/2,
        to_shape.top
    )
    connector.line.width = Inches(0.02)

# Connect main flow
connect(box1, box2)
connect(box2, box3)
connect(box3, box4)
connect(box4, box7)

file_path = "/mnt/data/給食業務システム概要図_1枚構成.pptx"
prs.save(file_path)

file_path
