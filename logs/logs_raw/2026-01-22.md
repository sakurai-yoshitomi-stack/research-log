## 1. 前提整理（ログの責務分離）

まず重要な前提。

### レイヤ別の責務
| 層 | 主な責務 | 操作者ログの主戦場 |
|---|---|---|
| React（フロント） | UI操作・入力 | ❌基本は保持しない |
| FastAPI（バック） | 認証・認可・業務処理 | ✅ **メイン** |
| PostgreSQL（DB） | データ永続化 | ✅ **補助（証跡）** |

👉 **操作者ログは DB ではなく API 層が主責務**  
DBは「最終証跡」を残す場所。

---

## 2. 「操作者ログ」の定義を明確化

### 操作者ログとは
> **誰が / いつ / どこから / 何を / どう変更したか**

最低限、以下6要素を揃える。

| 要素 | 内容 |
|---|---|
| operator_id | 操作者ID（ユーザーID） |
| operator_role | 権限（admin / user 等） |
| action | 操作種別（CREATE / UPDATE / DELETE / LOGIN 等） |
| target | 対象（テーブル名・画面ID・リソースID） |
| before | 変更前（差分 or 全体） |
| after | 変更後（差分 or 全体） |
| timestamp | 実行時刻 |
| source_ip | 接続元IP |
| trace_id | APIリクエスト追跡ID |

---

## 3. ログの種類を分ける（超重要）

### ① 業務操作ログ（必須）
- CRUD 操作
- ステータス変更
- 設定変更

👉 **API層で記録 → DB保存**

### ② 認証・認可ログ
- ログイン成功/失敗
- 権限エラー

👉 **API層**

### ③ DB操作証跡ログ（補助）
- 「実際にDBが変わった」証明
- APIを経由しない更新対策

👉 **DBトリガ or 拡張で最低限**

---

## 4. PostgreSQL 側：操作者ログ方針

### 4.1 DBに「操作者ログテーブル」を作る

```sql
CREATE TABLE operator_audit_log (
    id SERIAL PRIMARY KEY,
    operator_id VARCHAR(64),
    operator_role VARCHAR(32),
    action VARCHAR(32),
    target_table VARCHAR(64),
    target_id VARCHAR(64),
    before_data JSONB,
    after_data JSONB,
    executed_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    source_ip INET,
    trace_id UUID
);
```

ポイント：
- **JSONB**：差分でも全体でも対応可
- **trace_id**：APIログとの突合用

---

### 4.2 FastAPI → DB へ渡すべき情報

FastAPI で必ず Context として保持：

```python
request.state.operator_id
request.state.operator_role
request.state.source_ip
request.state.trace_id
```

👉 これを **全DB更新処理に必ず注入**

---

## 5. FastAPI 側：実装方針（超重要）

### 5.1 ミドルウェアで共通情報を確保

- trace_id を自動生成
- ユーザー情報を request.state に保存

```python
@app.middleware("http")
async def audit_context(request: Request, call_next):
    request.state.trace_id = uuid.uuid4()
    request.state.source_ip = request.client.host
    response = await call_next(request)
    return response
```

---

### 5.2 CRUD後に「必ず」ログを残す

**業務処理と分離しない（同一トランザクション）**

```python
async with db.transaction():
    before = await get_before_data()
    await update_data()
    after = await get_after_data()

    await insert_audit_log(
        operator_id,
        action="UPDATE",
        target="users",
        before=before,
        after=after
    )
```

👉 **業務がロールバックされたらログもロールバック**

---

## 6. DBトリガは「最小限」に使う

### 使う理由
- APIを通らない更新検知
- 不正操作の最終防衛線

### 使わない理由
- 操作者IDが取れない
- ログがブラックボックス化

### 方針
- **DELETE / UPDATE のみ**
- 操作者不明時は `operator_id = 'UNKNOWN'`

---

## 7. ログ保存・運用ポリシー

### 保存期間
| 種類 | 保持 |
|---|---|
| 操作者ログ | 3〜5年 |
| 認証失敗ログ | 1年 |
| アクセスログ | 3〜6ヶ月 |

### 改ざん防止
- UPDATE / DELETE 禁止
- 管理者でも論理削除のみ

```sql
REVOKE UPDATE, DELETE ON operator_audit_log FROM public;
```

---

## 8. React 側の方針（軽く）

- 画面操作ログは **送らない**
- 画面ID・操作種別のみAPIに渡す
- **操作者ログの真実はバックエンドで作る**

---

## 9. 方針まとめ（設計書用）

> 操作者ログは FastAPI 層を主とし、  
> PostgreSQL は改ざん不可な最終証跡として保持する。  
> 全操作は trace_id により API・DB ログを突合可能とする。  
> 操作者情報は DBトリガではなく API から明示的に注入する。

# 操作ログと PostgreSQL Auditログの境界整理（3階層構成）

## 1. 結論を先に（超重要）

> **操作者を“意味として”記録するのはアプリケーション（FastAPI）の責務**  
> **DBミドルのAuditログは「事実として起きたDB操作の証跡」**

つまり：

```
FastAPI操作ログ ＝ 意味・責任
PostgreSQL Auditログ ＝ 事実・証拠
```

この役割を混ぜると、だいたい破綻する。

---

## 2. PostgreSQLで取れるAuditログの正体

PostgreSQL（ミドル）で取れる代表例：

- logging_collector / log_statement
- pgaudit
- トリガベース監査

### PostgreSQL Auditログで分かること

| 分かる | 分からない |
|---|---|
| SQL文 | なぜその操作をしたか |
| 実行時刻 | 業務操作名 |
| DBユーザー | アプリ上の操作者 |
| 対象テーブル | 変更前後の意味 |

👉 **DBは「誰がログインした人間か」を知らない**  
知っているのは **接続ユーザー**だけ。

---

## 3. よくある勘違い（要注意）

❌「pgaudit入れてるから操作ログは十分」

→ **アウト**

理由：
- 全ユーザーが同一DBユーザー（例：app_user）
- Reactの操作者IDはDBに来ない
- 監査で「この更新は誰？」に答えられない

---

## 4. 境界線の定義（ここが核心）

### 4.1 FastAPI（アプリ層）の責務

**操作者ログ（主ログ）**

> 「誰が、どの権限で、何という業務操作を行い、  
> 結果としてどのデータがどう変わったか」

#### 具体項目
- operator_id（ユーザーID）
- role / 権限
- action（業務操作名）
- target（業務的対象）
- before / after
- trace_id

👉 **監査・問い合わせ・責任追跡はここを見る**

---

### 4.2 PostgreSQL（DBミドル）の責務

**Auditログ（副ログ）**

> 「DBとして、どのSQLが、いつ、どの接続で実行されたか」

#### 具体項目
- SQL文
- テーブル名
- 実行時刻
- DBユーザー
- セッションID

👉 **改ざん検知・API非経由操作の発見用**

---

## 5. 境界を表にするとこう

| 観点 | FastAPI 操作ログ | PostgreSQL Audit |
|---|---|---|
| 操作者特定 | ◎（ユーザーID） | △（DBユーザーのみ） |
| 業務意味 | ◎ | × |
| 変更前後 | ◎ | △（SQL解析次第） |
| 改ざん検知 | △ | ◎ |
| API非経由検知 | × | ◎ |
| 監査向き | ◎ | 補助 |

---

## 6. 正しい「合わせ技」構成

### 6.1 trace_id で紐付ける

FastAPI → DB接続時に：

```sql
SET application_name = 'trace_id=xxxx';
```

または：

```sql
SET SESSION "trace_id" = 'xxxx';
```

👉  
- API操作ログ  
- PostgreSQL Auditログ  

を **同一trace_idで突合**できる。

---

### 6.2 DBトリガの立ち位置

DBトリガは **例外処理用**。

#### 使う場合
- DELETE / UPDATE のみ
- operator_id が渡っていれば記録
- 渡っていなければ `UNKNOWN`

#### 使わない理由
- 業務意味が分からない
- ロジックがブラックボックス化

---

## 7. 設計書にそのまま書ける整理文

### 方針文（おすすめ）

> 本システムでは、  
> 操作者を特定可能な操作ログはアプリケーション層（FastAPI）で記録する。  
> PostgreSQLのAuditログは、APIを経由しないDB操作および改ざん検知の補助証跡として利用する。  
> 両ログは trace_id により相互参照可能とする。

---

## 8. まとめ（超短縮）

- ✔ **単純に言えば両方とも「操作ログ」**
- ✔ でも  
  - **FastAPI：責任・意味**  
  - **PostgreSQL：事実・証拠**
- ✔ DB Auditだけで完結させようとしない
- ✔ 境界を決めて「合わせ技」にする

- 差分設計が複雑

### 3. 監査要件がそこまで厳しくないケースが多い

# 別紙：操作ログ／監査ログ設計の検討背景および採用理由

## 1. 背景

本システムは以下の構成を採用している。

- フロントエンド：React  
- バックエンド：FastAPI  
- データベース：PostgreSQL  

業務上、  
「誰が・いつ・どのデータを更新したか」を  
**監査・障害調査・問い合わせ対応で追跡可能とすること**が求められる。

一方で、
- パフォーマンス
- 実装・運用コスト
- バルク更新の発生

を考慮し、  
**すべての更新について変更前後を厳密に取得する方式は現実的でない**。

そのため、複数の方式を比較検討した。

---

## 2. 検討した方式

### 案① PostgreSQL Auditログ（pgaudit 等）のみで対応

**概要**  
PostgreSQLのAudit機能によりSQL実行履歴を取得する。

**課題**
- DBユーザー単位の記録となり、業務ユーザーを特定できない
- 操作の業務的意味（画面・機能）が分からない
- 監査時に「誰がやったか」を説明できない

**結論**  
監査要件を満たさないため不採用。

---

### 案② UPDATE前にSELECTを行い、変更前後を全件保持

**概要**  
UPDATE前に対象レコードをSELECTし、before / after を操作ログとして保存。

**課題**
- SELECT + UPDATE による性能劣化
- バルク更新時の実装・ログ量が過大
- 全操作に適用するのは運用上困難

**結論**  
一部重要機能では有効だが、全体方式としては不採用。

---

### 案③ DBトリガーで変更前後を完全取得

**概要**  
PostgreSQLトリガーにより OLD / NEW を監査ログに保存。

**課題**
- 業務ユーザーID・操作理由を取得できない
- 業務ロジックがDBに分散し、ブラックボックス化
- 保守性・可読性が低下

**結論**  
証跡用途としては有効だが、主方式としては不適。

---

## 3. 採用方式（本システム方式）

### 概要

以下の方式を採用する。

1. 各業務テーブルに  
   - 業務ユーザーID（updated_by）  
   - 更新日時（updated_at）  
   を必須項目として保持する
2. FastAPI が更新時に上記項目を必ず設定する
3. PostgreSQLトリガーにより、更新後状態を監査ログとして保存する
4. 操作ログ（FastAPI）とDB監査ログは  
   業務ユーザーIDおよび更新時刻によりひも付ける

---

## 4. 採用理由

### ① 業務ユーザーを確実に特定できる

- DBユーザーに依存せず
- アプリケーション上の操作者を直接記録可能

---

### ② パフォーマンスと実装負荷のバランスが良い

- UPDATE前SELECTが不要
- バルク更新にも対応可能
- 実装が単純で保守しやすい

---

### ③ 監査上、十分な追跡性を確保できる

- 「誰が・いつ・どのレコードを更新したか」を追跡可能
- 操作ログ（FastAPI）とDB監査ログの突合が可能
- API非経由の更新も検知可能

---

### ④ 不正・想定外操作の検知が可能

- updated_by / updated_at が未設定の場合、
  API非経由操作として検知可能
- 監査観点での抑止力を持つ

---

## 5. 留意事項（補足）

- 操作理由・業務操作名はFastAPI側の操作ログで管理する
- 変更前状態が必須な業務については、個別に例外対応を検討する
- updated_by / updated_at は NOT NULL 制約を設け、必須入力とする

---

本方式は、

- 監査要件
- 性能
- 実装・運用負荷
